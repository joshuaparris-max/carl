From Python prototype to AAA-indie — solo dev roadmap

Four pillars, one developer, sequenced by what ships fastest and matters most.

The hard truth first. "All four pillars at once" is how solo devs ship nothing. The good news: these four pillars have a natural build order where each one makes the next easier. Follow the sequence in the last tab and you will ship a demo that feels AAA-indie within 3–4 months of consistent work.
What you already have

Genuinely strong foundation

  • Data-driven content in JSON — critical for scaling
  • Separated systems (world, state, combat, quests) — clean to port
  • Working save/checkpoint — most devs skip this and regret it
  • Unit tests passing — extremely rare at this stage
  • All 7 tutorial quests driving the opening loop
What's blocking AAA feel

The three actual gaps

  • No visual renderer — the terminal is a prototype shell, not a game
  • No audio — silence is the single biggest quality signal players use
  • No "juice" — hit effects, screen shake, particle bursts, transitions
  • No real UI system — health bars, quest log, dialogue box need design
  • No main menu / loading / game-over screens
Current prototype

Playable terminal demo status

  • Core loop is live: `crawler.py` starts the game, `src/game.py` runs the turn loop and UI render.
  • Working actions: movement, talk, open, inventory, craft, attack, use, equip, pickup, shop, companion modes, save/load/checkpoint.
  • Quest progression and completion notifications now display immediately during play.
  • World state includes map rendering, entity positions, objects, hazards, and enemy turns.
  • Unit tests for the core prototype are passing with `unittest discover tests`.
The leverage point: Port your Python logic to JavaScript first (it maps almost 1:1), then build the visual layer on top. Your JSON data files, quest structure, and combat math all survive the port unchanged. You are not starting over — you are adding a face to what already works.

The single most important decision. Get this wrong and you fight your tools forever.

Recommendation for this project: Phaser 3 + web, packaged with Electron/Tauri for desktop. Here's why every other option loses for your specific situation.
Recommended

Phaser 3 (web-first, JavaScript)

  • Your existing JS game logic ports directly — same language, same data structures
  • Built-in tilemap rendering (Tiled .tmx format, which your map already almost matches)
  • Scene system handles main menu, game, pause, game-over as separate scenes
  • Web Audio API integration built in — sound is one require() call away
  • Particle system, tweens, camera shake all built in — juice is free
  • Package with Electron or Tauri for Windows/Mac desktop builds, zero code changes
  • Active community, excellent docs, MIT license
phaser.iofree & open sourceweb + desktopyour code language
Skip

Godot 4

Excellent engine but forces a Python→GDScript rewrite of all your logic. GDScript is similar but not the same — every function, class, and JSON parser needs manual conversion. Adds 4–6 weeks of rework for no gameplay benefit.

Skip

Unity

C# means a full rewrite. Overkill for a tile-based dungeon crawler. The license changes of 2023 make it a poor choice for a solo dev building to distribute. Phaser does everything you need without the overhead.

Skip

Pure HTML/JS

Fine for the prototype phase (which you've already proven). Does not scale to a real game — no scene management, no asset pipeline, no audio system, no tilemaps. You'll rebuild Phaser badly. Use the real thing.

The visual identity of this game is already defined by the books. Dark, grimy dungeon with flashes of absurdist brightness (Donut's fur, explosion effects, System announcements in garish amber). Lean into that contrast — it's your aesthetic.
Art style decision

16×16 pixel art — the right call

  • Achievable solo without hiring an artist
  • Tilesets for dungeon environments exist free (Kenney.nl, itch.io) as a starting point
  • 16px tiles on a 12×12 grid = 192×192 game viewport — scales beautifully to any screen
  • Character sprites: Carl (boxer shorts, jacket), Donut (fluffy Persian), Mordecai (Chupacabra) — each 16×16, 4-frame walk cycle minimum
  • Free tools: Aseprite ($20, worth every cent), or Libresprite (free fork)
Atmosphere layer

What makes it look premium

  • Animated tiles: flickering torches, dripping water, chest glow
  • Parallax background behind the tilemap — subtle depth
  • Post-processing: a very slight vignette and scanline overlay (pure CSS, no render cost)
  • Color palette: 32-color limit forces coherence — deep stone grays, amber fire, teal for Donut moments, red for System danger alerts
  • Day/night system per floor (Floor 1 savannas are bright; deeper floors get darker)
Immediate wins (this week)

Before drawing a single pixel

  • Download the free Kenney Tiny Dungeon tileset — it covers walls, floors, chests, NPCs, enemies. Placeholder for everything while you build systems.
  • Set up Phaser 3 with a Tiled tilemap — your existing map.txt converts to Tiled in under an hour
  • Add a CRT/vignette CSS overlay to the canvas — takes 10 lines and immediately makes it look intentional
  • Use Phaser's built-in tween system for a simple tile pulse on the player position — makes it feel alive instantly
Game feel is the highest return-on-investment work in game dev. One afternoon of "juice" makes players think the whole game is more polished than it is. This is not optional — it is what separates indie from AAA-indie.
Combat juice — do these first

Impact feedback

  • Screen shake on every hit — Phaser camera.shake(), 2 lines of code
  • Hit flash: enemy sprite flashes white for 80ms on damage received
  • Damage number floats up from the hit position and fades out
  • Death animation: enemy sprite scales up briefly then disappears (tween)
  • Carl's attacks: brief forward lunge animation before the hit resolves
Movement juice — makes exploration satisfying

Locomotion feel

  • Smooth movement: instead of snapping tile-to-tile, tween the player over 80–100ms — makes grid movement feel fluid not janky
  • Donut's follow: she moves 1 frame after Carl with a slight ease-out — feels organic
  • Footstep sounds timed to movement (one per tile crossed)
  • Tile landing: a tiny scale bounce (1.0 → 1.05 → 1.0) on the player sprite each step
UI juice — quest & System

Feedback moments

  • Quest complete: amber banner slides in from top, holds 2s, fades out
  • System announcements: text types out character-by-character with a blinking cursor
  • Level up: brief full-screen amber flash at 20% opacity with a triumphant sound
  • Inventory open: panel slides in from the right with ease-out (CSS transform)
  • New item found: glow pulse on the item tile before pickup
Audio — the most underrated pillar

Sound design priorities

  • Footsteps (stone floor sound, different for Carl vs Donut)
  • Attack swing + hit impact (two separate sounds — swing then thud)
  • Explosion (this is Carl's signature — make it satisfying)
  • Chest open (satisfying click + shimmer)
  • System announcement jingle (brief alien-sounding motif)
  • Ambient dungeon loop (distant drips, low rumble)
  • Source: freesound.org, Kenney audio packs — all free
Content depth is the last pillar to build, not the first. A game with great feel and no content is a tech demo. A game with bad feel and great content is unplayable. But if feel is there, even modest content can carry a full game.
Tutorial Guild (done)

Depth it still needs

  • Mordecai's 3-dialogue cycle → expand to 8–10 lines that escalate in stress
  • Donut's ambient commentary per tile type (she hates the water tiles, loves the shop)
  • Tutorial chest should have a cinematic open sequence, not just a message
  • Add a second NPC in the guild — a fellow crawler who gives bad advice
  • System announcements on every quest completion, not just first ones
Floor 1 — build this next

Minimum viable first floor

  • 4×3 rooms connected by corridors (48 tiles wide total) — manageable scope
  • 3 enemy types: training goblin, patrol kobold, sleeping crawler-turned-enemy
  • 1 mini-boss room with a named enemy and a pre-fight System announcement
  • 2 saferoom doors (one locked, one open) — teach the saferoom mechanic
  • The collapse timer becomes real here — 200-turn hard deadline
Crafting — Carl's identity system

What it needs to feel right

  • Discovery-based recipes: Carl figures them out by combining items, not reading a menu. First time he crafts an explosive, the System mocks him for discovering it by accident.
  • 3 starting recipes accessible from tutorial materials
  • Crafting UI: a simple 2-slot combine interface with a "Try it" button — no recipe book initially
  • Failed crafts produce a flavorful failure message from the System, not silence
  • Explosion radius visualized on the grid before throwing — tactical preview
Polish is not decoration — it is information design. Every screen transition, loading state, and error message is a communication to the player. AAA-indie games feel polished because every state is handled intentionally.
Screens you need

The full screen inventory

  • Title screen — dark, atmospheric, Donut visible in the corner judging you
  • New game / Continue / Options — minimal menu, keyboard navigable
  • Tutorial intro — System announcement that plays before control is given
  • Pause screen — the dungeon freezes, System makes a comment about stopping
  • Game over — death screen with cause of death and the System's commentary on it
  • Floor transition — countdown + "Entering Floor X" animation
  • Victory screen (tutorial complete) — viewer count reveal, quest summary
UX improvements to existing systems

What the prototype teaches wrong

  • Replace the keyboard shortcut wall with a contextual controls bar — show only relevant actions for what Carl is adjacent to
  • Quest log needs a collapsible UI, not a raw text dump
  • Inventory should show item stats on hover before equipping
  • Dialogue needs a speaker portrait, not just a name prefix
  • Save should auto-happen on saferoom entry, not require a keypress
  • The "No Pants" debuff should have a visible indicator on Carl's sprite, not just a stat line
Accessibility (don't skip this)

Minimum viable accessibility for web release

  • Colorblind mode: all color-coded elements (enemy HP, terrain type) get a secondary shape indicator
  • Text size: UI text should scale with browser font settings
  • Reduced motion: all tweens and shakes should be skippable via a toggle
  • Keyboard-only: every action must be reachable without mouse — this is already true but needs testing
  • Pause on focus loss: browser tab switching should auto-pause the turn timer
Do not work on all four pillars simultaneously. The sequence below is designed so each week produces something playable and demoable. If you follow it, you will have a shippable demo in 12–14 weeks of consistent solo work.
Weeks 1–2 — engine port

Port Python logic to Phaser 3

  • Set up Phaser 3 project with Vite (fast dev server, easy Electron packaging later)
  • Port game_state.js, world.js, combat.js, quests.js — your Python maps almost directly
  • Load existing JSON data files (characters, items, dialogue, recipes) unchanged
  • Load tutorial_guild_map.txt as a Tiled tilemap
  • Placeholder graphics only (Kenney Tiny Dungeon) — do not make art yet
  • Milestone: the tutorial guild is playable in a browser window with placeholder art
Weeks 3–4 — juice pass

Game feel before any new content

  • Smooth movement tweens (80ms per tile)
  • Screen shake + hit flash + floating damage numbers
  • System announcement typewriter effect
  • Quest complete banner animation
  • Ambient sound loop + footsteps + one explosion sound
  • Milestone: show someone the game — they should say "this feels good" before asking about content
Weeks 5–7 — art pass

Replace placeholder art with custom pixel art

  • Carl sprite: 16×16, 4-direction walk cycle (4 frames each), idle, attack
  • Donut sprite: 16×16, walk cycle, idle (make her fluffy and proud)
  • Mordecai sprite: 16×16, Chupacabra form, stressed idle animation
  • Tileset: dungeon walls, floor, desk, chest, hazard tiles (custom or modified Kenney)
  • UI panels: health bars, quest log, dialogue box — styled to the dark/amber aesthetic
  • Milestone: screenshot looks like a game you'd pay for
Weeks 8–10 — Floor 1 content

The first full floor

  • Design and build Floor 1 map (4×3 rooms, corridors, saferooms)
  • 3 enemy types with distinct behavior (patrol, stationary, reactive)
  • Crafting UI with discovery-based recipes
  • The collapse timer as a visible, ticking, increasingly dramatic countdown
  • Floor transition screen with System commentary
  • Milestone: a complete play loop — tutorial → Floor 1 → death or completion
Weeks 11–12 — polish & screens

Every screen handled, no rough edges

  • Title screen, main menu, game over, pause screen
  • Auto-save on saferoom entry
  • Contextual controls bar replacing the keyboard wall
  • Accessibility: colorblind mode, reduced motion toggle
  • Electron packaging for desktop builds (add this last — it is 1 config file)
  • Milestone: this is your itch.io demo release
